## Summary

All products with a discount are showing **$0.00** as their sale price on the homepage and product detail pages. This affects 3 out of 4 products in the catalog.

---

## Steps to Reproduce

1. Open http://localhost:5000
2. Observe all discounted products show **$0.00** sale price
3. Example: Laptop Pro 15 (10% off $1,299.99) shows **$0.00** instead of **$1,169.99**

---

## Expected Behavior

Correct discounted prices shown:
- Laptop Pro 15 (10% off): **$1,169.99**
- Wireless Headphones (20% off): **$159.99**
- Mechanical Keyboard (15% off): **$127.49**

---

## Root Cause

A refactor in `api-service` renamed the JSON response field:

```
Before: { "sale_price": 1169.99 }   ← frontend expects this
After:  { "final_price": 1169.99 }  ← api-service now returns this
```

The frontend (`test-webapp`) was not updated and still reads `sale_price`, which is now missing from the API response. Python's `.get("sale_price", 0.0)` silently returns `0.0` instead of raising an error — so the bug is silent in logs.

---

## Suggested Fix

**Option A — Revert API field name (less disruptive):**

In `api-service/api.py`, rename `final_price` back to `sale_price`:
```python
# Before (buggy)
product["final_price"] = compute_sale_price(p["price"], p["discount"])

# After (fixed)
product["sale_price"] = compute_sale_price(p["price"], p["discount"])
```

**Option B — Update frontend to match new field name:**

In `test-webapp/app.py`, update both multi-repo functions to read `final_price`:
```python
p["sale_price"] = p.get("final_price", 0.0)
```

Option A is preferred — `sale_price` is more semantically clear and is already the established contract.
